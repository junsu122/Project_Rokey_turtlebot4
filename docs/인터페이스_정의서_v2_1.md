## 0. 설계 원칙 (소유권 원칙)

1. **Robot State는 로봇이 소유한다.** 모든 상태 전이는 로봇이 스스로 수행하고 IF-02로 보고한다. FMS는 상태를 명령하지 않는다.
2. **Mission State는 FMS가 단독 소유한다.** 로봇은 Mission State를 보고하지도, 알지도 않는다. FMS가 로봇들의 IF-02 보고를 근거로 계산한다.
3. **명령은 task로만 한다.** FMS는 "무엇을 하라"(task)를 내리고, 로봇은 "무엇을 하고 있다"(state)와 "task가 어떻게 됐다"(task_status)를 보고한다. 전이의 계기가 task 수신/완료일 뿐, 전이 자체는 항상 로봇의 것이다.
4. **순찰은 task가 아니라 기본 동작(default behavior)이다.** 로봇은 활성 task가 없으면 스스로 PATROL로 전환한다. FMS는 지시하지 않고 IF-02로 관측만 한다. 순찰 중 `current_task_id = null`.
5. **로봇 유닛 간 직접 통신은 없다.** 모든 메시지는 FMS를 경유한다.
6. **공통 필드:** 모든 메시지에 `timestamp`(ISO 8601), `msg_id`, `version` 포함.

> driving 트랙 계약 한 줄 요약: **"task가 있으면 task를 수행하고, 없으면 순찰한다. 어느 쪽이든 상태는 IF-02로 보고한다."**

## 1. 상태 모델 정의

### 1.1 Robot State (로봇 소유, IF-02로 보고)

| State | 의미 | 비고 |
|---|---|---|
| IDLE | 대기 중 | 임무 없이 정지 |
| PATROL | 순찰/탐지 모드 | **기본 동작.** task 없을 때 자율 진입 |
| INTERACTING | 사용자 응대 중 | 호출받고 정지, 목적지 입력/확인 |
| RESERVED | 임무 예약됨 | next_robot이 MOVE_TO_STANDBY 수행 중 |
| HANDOVER_READY | 인수 준비 완료 | next_robot이 대기지점 도착 후 대기 |
| ESCORTING | 사용자 안내 중 | start/next 공통 |
| WAITING_HANDOVER | 핸드오버 지점 대기 | start_robot이 연결지점 도착 후 |
| RETURNING | 복귀 중 | RETURN_TO_BASE 수행 중 |
| EMERGENCY | 비상 상태 | 범퍼 감지, 비상정지 |
| ERROR | 오류 상태 | 통신 장애, 위치 추정 실패 |

※ CHARGING은 스코프 아웃 (IF-02 battery 필드만 유지, 필요 시 추가)

### 1.2 Mission State (FMS 단독 소유, 로봇은 모름)

```
REQUESTED → ASSIGNED → ESCORTING_TO_HANDOVER → HANDOVER_WAITING
          → ESCORTING_TO_FINAL → COMPLETED
(예외: CANCELLED / EMERGENCY / FAILED)
```

- mission은 **목적지 확정(IF-01 수신) 시 생성**된다. 호출·대화 단계는 로봇의 로컬 상황(Robot State = INTERACTING)일 뿐 mission이 아니다.
- PATROL은 mission이 아니다. "활성 mission 없음"은 미션 테이블이 아니라 Robot State(PATROL)에서 읽는다.
- 같은 층 에스코트: ASSIGNED → ESCORTING_TO_FINAL 직행 (핸드오버 단계 생략).

## 2. IF-01. 고객 요청 (Interaction → FMS)

**목적:** Interaction 트랙이 파악한 고객 요구를 FMS에 전달. Interaction의 책임은 전달까지, 판단·할당은 FMS.

```json
{
  "msg_id": "...", "version": "2.0",
  "request_id": "REQ_0012",
  "robot_id": "robot2",
  "request_type": "ESCORT",
  "destination": { "poi_id": "GATE_30", "floor": 2 },
  "origin": { "floor": 1, "pose": { "x": 0.0, "y": 0.0 } },
  "customer": { "customer_id": "C_xxx", "profile": "ELDERLY", "language": "ko" },
  "timestamp": "..."
}
```

| 필드 | 설명 |
|---|---|
| request_type | `ESCORT` \| `CANCEL` |
| target_request_id | (CANCEL 시) 취소할 원 요청 ID |
| destination.poi_id | LLM이 자유 발화를 정규화한 POI ID. 좌표/층 조회는 FMS POI 테이블 담당 |
| customer.profile | GENERAL \| ELDERLY \| FOREIGNER \| VISUALLY_IMPAIRED — 주행 파라미터에 반영 |
| customer.language | 도착 측 로봇 인사말(Stage 4)용 |

- **IF-01은 유효한 목적지가 확정된 경우에만 발행된다** (2.1절). FMS는 IF-01의 poi_id가 항상 유효하다고 신뢰해도 된다.

### 2.1 목적지 유효성 검증 (Interaction 내부 처리)

발행 전제조건: **destination.poi_id는 POI 목록에 존재하는 유효한 ID여야 한다.** 검증 책임은 Interaction 노드에 있다.

- **POI 목록 배포:** 각 로봇 탑재 노트북에 정적 파일(예: `poi_table.yaml`)로 배포한다. FMS의 POI 테이블과 동일한 원본에서 생성하며, 맵 변경 시 함께 갱신한다. 파일에는 `poi_id`, 표시명(한/영), 층, 동의어 목록(예: "게이트 삼십", "30번 게이트")을 포함한다.
- **검증 흐름:** 고객 발화 → LLM이 POI 후보로 정규화 → POI 목록 대조 → (성공) 고객에게 목적지 확인 후 IF-01 발행 / (실패) IF-01을 발행하지 않고 재질문 ("그 위치는 안내가 어려워요. 혹시 ○○ 말씀이신가요?"). 재질문 N회(권장 2회) 실패 시 정중히 종료하고 INTERACTING → PATROL/IDLE 복귀.
- **효과:** 구 예외 전이 6(목적지 해석 실패)은 FMS에 도달하지 않는다. FMS의 예외 표에서 제거(7장).

### 2.2 FMS 무응답 감지 — Interaction 로컬 타임아웃 (예외처리 구현 최우선)

Wi-Fi는 안정적이라고 가정하지만, FMS 프로세스 다운·브로커 장애 등 "요청이 접수되지 않는 상황"은 시연 중에도 발생할 수 있으므로 **예외처리 중 최우선으로 구현한다.** 신규 인터페이스 없이 Interaction 트랙 내부에서 처리한다.

- **원리:** FMS가 IF-01을 정상 처리하면 수 초 내 ESCORT_TO_HANDOVER(또는 같은 층이면 ESCORT_TO_FINAL) task가 도착하여 자기 로봇이 INTERACTING → ESCORTING으로 전이한다. Interaction 노드는 같은 노트북·같은 ROS 도메인이므로 이 전이를 로컬 토픽으로 직접 관찰할 수 있다.
- **동작:** IF-01 발행 시점에 타이머 시작(권장 5초) → 시간 내 자기 로봇의 ESCORTING 전이 확인 시 타이머 해제, 안내 시작 멘트 → 미확인 시 고객에게 "지금 안내 시스템 연결이 원활하지 않아요. 잠시 후 다시 말씀해 주세요" 안내 후 INTERACTING 종료, 동일 request_id 재사용 금지(재시도 시 새 request_id).
- **주의:** FMS가 뒤늦게 응답해 task가 타임아웃 직후 도착할 수 있다. 로봇(driving)은 자신이 INTERACTING/ESCORTING이 아닌 상태에서 받은 ESCORT 계열 task를 거절(ACK=REJECT)한다 — 고객이 이미 떠난 뒤의 유령 에스코트 방지.

## 3. IF-02. 로봇 상태 보고 (Robot → FMS)

**목적:** 로봇의 현재 상태 + 수행 중 task의 결과를 FMS에 보고. **task 결과 보고 채널을 겸한다** (별도 결과 메시지 없음). 순찰 중 좌표 관측도 이 채널 하나로 충분하다.

```json
{
  "msg_id": "...", "version": "2.0",
  "robot_id": "robot2",
  "state": "ESCORTING",
  "pose": { "x": 0.0, "y": 0.0, "theta": 0.0 },
  "battery": 78,
  "current_task_id": "TASK_0034",
  "task_status": "RUNNING",
  "error_code": null,
  "timestamp": "..."
}
```

| 필드 | 설명 |
|---|---|
| state | 1.1의 Robot State enum |
| current_task_id | 수행 중 task. **null = 기본 동작(순찰/대기) 중** |
| task_status | `ACCEPTED` \| `RUNNING` \| `SUCCEEDED` \| `FAILED` \| null |

**발행 규칙:** 1~2Hz 주기 발행 + **상태 전이 및 task_status 변화 시 즉시 발행** (핸드오버 3초 측정의 전제).

## 4. IF-03. 임무 할당 (FMS → Robot)

**목적:** FMS가 전이표에 따라 로봇에 task를 발행. 로봇이 외부에서 받는 **유일한 입력 채널.**

```json
{
  "msg_id": "...", "version": "2.0",
  "task_id": "TASK_0034",
  "robot_id": "robot4",
  "task_type": "MOVE_TO_STANDBY",
  "goal": { "poi_id": "ES_03", "pose": { "x": 0.0, "y": 0.0 }, "floor": 2 },
  "customer": { "customer_id": "C_xxx", "profile": "ELDERLY", "language": "ko" },
  "mission_id": "MISSION_0012",
  "cancel_task_id": null,
  "timestamp": "..."
}
```

**task_type (4종):**

| task_type | 의미 | 도착 후 |
|---|---|---|
| MOVE_TO_STANDBY | 고객 없이 지정 지점으로 이동해 **대기** | HANDOVER_READY로 후속 task 대기 (mission에 묶임) |
| ESCORT_TO_HANDOVER | 고객 동반, 핸드오버 지점까지 안내 | WAITING_HANDOVER로 대기 |
| ESCORT_TO_FINAL | 고객 동반, 최종 목적지까지 안내 | task 종료 (mission 완료 보고) |
| RETURN_TO_BASE | 임무 종료 후 복귀 | task **완전 종료** → 기본 동작(순찰) 복귀 |

- task는 **역할 중립**으로 정의한다. MOVE_TO_STANDBY는 현재 FSM에서 next_robot만 받지만, "누가 받느냐"는 FMS 발행 로직의 몫이고 task 의미는 일반 동작이다.
- MOVE_TO_STANDBY vs RETURN_TO_BASE 구분 기준: **도착 후 "기다리느냐(전자), 끝나느냐(후자)"**.
- 취소: `cancel_task_id`에 취소 대상 task_id를 담아 발행 → 로봇은 해당 task 중단 후 후속 task(보통 RETURN_TO_BASE) 수행.
- **응답:** 로봇은 ACK(수락/거절) 반환. 이후 진행 상황은 IF-02 task_status로 보고.

## 5. IF-05. 이상 상황 알림 (Robot → FMS/관제)

**목적:** 순찰 중 로봇 탑재 카메라 CV(YOLO)가 감지한 이상 상황을 즉시 전송 (BRD 3.4). 외부 CCTV 아님 — 서버팀 설계서 5-a-v와 일치.

```json
{
  "msg_id": "...", "version": "2.0",
  "event_type": "FIRE",
  "robot_id": "robot2",
  "confidence": 0.91,
  "location": { "x": 0.0, "y": 0.0, "floor": 1 },
  "snapshot_ref": "img_20260606_1432.jpg",
  "timestamp": "..."
}
```

**event_type:** `FIRE` | `SUSPICIOUS_PERSON`

## 6. (구 IF-04) 핸드오버 표현 — 인터페이스 삭제, 매핑으로 대체

핸드오버 전용 메시지는 없다. 모든 핸드오버 이벤트는 IF-02 전이와 IF-03 발행으로 표현된다:

| v1 IF-04 이벤트 | v2 표현 |
|---|---|
| HANDOVER_READY (robot2 도착) | robot2의 IF-02: ESCORTING → WAITING_HANDOVER (task SUCCEEDED) |
| PICKUP_ACTIVATE | FMS의 IF-03: ESCORT_TO_FINAL → robot4 |
| PICKUP_WAITING (robot4 준비) | robot4의 IF-02: RESERVED → HANDOVER_READY (task SUCCEEDED) |
| CUSTOMER_DETECTED | **스코프 아웃** (Q4) — 핸드오버 승인만으로 robot4가 안내 시작. 위험 2는 robot4의 인사 멘트 + 얼굴 UI로 완화 |
| handover_id | 불필요 — mission_id로 충분 |

**핸드오버 승인 조건 (FMS):** `robot2.state == WAITING_HANDOVER ∧ robot4.state == HANDOVER_READY` — 어느 쪽이 먼저 와도 동작 (예외 전이 1, 2 통합).

**수락 기준 "3초" 측정:** robot2의 WAITING_HANDOVER 전이 보고 timestamp ↔ FMS의 ESCORT_TO_FINAL task 발행 timestamp 차이. (전 기기 NTP 동기화 전제)

## 7. 상태 전이 ↔ 인터페이스 매핑 (정상 흐름)

서버팀 전이표를 v2 의미로 재구성. "Robot State 변화"는 FMS가 명령하는 게 아니라 **기대 전이**(IF-02로 검증, 어긋나면 이상 감지)다.

| No | Trigger (인터페이스) | Mission State | 기대 Robot State | FMS 처리 (IF-03 발행) |
|---|---|---|---|---|
| 1 | 사용자 호출 (로봇 로컬) | — (mission 없음) | robot2: PATROL/IDLE → INTERACTING | 없음 (IF-02 관측만) |
| 2 | robot2의 IF-01 ESCORT | **mission 생성** → REQUESTED | robot2: INTERACTING 유지 | 목적지/구역 저장 |
| 3 | 로봇 배정 완료 | REQUESTED → ASSIGNED | robot4: PATROL/IDLE → RESERVED | **MOVE_TO_STANDBY → robot4** |
| 4 | 배정 직후 | ASSIGNED → ESCORTING_TO_HANDOVER | robot2: INTERACTING → ESCORTING | **ESCORT_TO_HANDOVER → robot2** |
| 5 | robot4의 IF-02 (task SUCCEEDED) | 변화 없음 | robot4: RESERVED → HANDOVER_READY | 준비 완료 기록 |
| 6 | robot2의 IF-02 (task SUCCEEDED) | ESCORTING_TO_HANDOVER → HANDOVER_WAITING | robot2: ESCORTING → WAITING_HANDOVER | 도착 이벤트 저장, **3초 측정 시작** |
| 7 | 승인 조건 충족 (5∧6) | HANDOVER_WAITING → ESCORTING_TO_FINAL | robot4: HANDOVER_READY → ESCORTING / robot2: WAITING_HANDOVER → RETURNING | **ESCORT_TO_FINAL → robot4**, **RETURN_TO_BASE → robot2** |
| 8 | robot2의 IF-02 (task SUCCEEDED) | 변화 없음 | robot2: RETURNING → PATROL (기본 동작) | 상태 갱신 |
| 9 | robot4의 IF-02 (task SUCCEEDED) | ESCORTING_TO_FINAL → COMPLETED | robot4: ESCORTING → RETURNING | 도착 저장, **RETURN_TO_BASE → robot4** |
| 10 | robot4의 IF-02 (task SUCCEEDED) | (COMPLETED 유지, 로그 저장) | robot4: RETURNING → PATROL | mission 종료 로그 |
| 11 | (같은 층) 배정 시 next 불필요 | ASSIGNED → ESCORTING_TO_FINAL | robot2: INTERACTING → ESCORTING | **ESCORT_TO_FINAL → robot2** (3, 5~8 생략) |

**예외 매핑 요약:**

| 상황 | 감지 채널 | Mission | FMS 처리 |
|---|---|---|---|

## 9. 통신 구조 (구조 확정 · 전송 프로토콜 미확정)

**확정된 것 (구조):**
- 로봇 유닛 = TurtleBot4(RPi4) + 탑재 노트북, 유닛별 ROS_DOMAIN_ID 분리 (robot2=1, robot4=2). ROS/DDS는 유닛 내부 전용.
- 로봇 ↔ FMS: **비-ROS JSON 채널** (노트북의 브리지 노드가 ROS 토픽 ↔ JSON 변환). FMS는 ROS 비의존.
- 관제 UI ↔ FMS: Flask REST (읽기 전용 조회).
- FMS 내부 모듈: Flask API + 로봇 통신 클라이언트 + Mission Manager + State Machine(전이표) + DB Manager(SQLite).
- 운영 전제: 전 기기 고정 IP + 동일 서브넷, FMS PC를 NTP(chrony) 서버로 시간 동기화.

**미확정 (후보: MQTT 권장, WebSocket 가능):** 로봇↔FMS 채널의 구체 프로토콜. MQTT 권장 사유 — push 지원, pub/sub 모델이 IF 구조와 일치, 브로커(Mosquitto) 설치·paho-mqtt 구현 부담 적음.

### 9.1 전송 프로토콜 결정 시 함께 닫아야 할 사항 (구 "비어 있는 것 4번")

이 항목들은 우리 구조(도메인 분리 + 비-ROS 채널)가 필연적으로 만드는 고려사항이며, 프로토콜이 정해지는 즉시 같이 결정한다:

| # | 항목 | 내용 | 프로토콜 의존성 |
|---|---|---|---|
| 1 | 배달 보장 수준 | 메시지별 유실 허용 여부. IF-02(주기 보고)=유실 허용, **IF-01/03/05=보장 필수** (MQTT면 QoS 0/1, WebSocket이면 앱 레벨 ACK+재전송) | 프로토콜별 수단 다름 |
| 2 | 중복 처리 (멱등) | 보장 배달은 중복 도착 가능 → **로봇은 이미 처리한 task_id 재수신 시 무시.** | **프로토콜 무관, driving 계약에 항상 포함** |
| 3 | ACK 응답 경로 | IF-03 ACK가 돌아올 채널과 task_id 상관 방법 (MQTT면 `robot/{id}/task_ack` 토픽, WebSocket이면 양방향이라 자연 해결) | 프로토콜별 설계 다름 |
| 4 | 채널 네이밍 | 토픽/엔드포인트 명명 규칙. MQTT 채택 시 권장: `robot/{id}/status·task·task_ack·request·event` | 프로토콜 확정 후 |

스코프 축소 전제(동시 호출 배제, Wi-Fi 안정 가정) 하에서 ACK 타임아웃·재전송 정책은 생략 가능 — FMS 무응답은 2.2절 로컬 타임아웃이 최후 방어선.

## 부록 C. 스코프 아웃 기록 — 배정 실패 응답 (사업화 시 복원용)

> 이 부록은 "왜 뺐는지"를 잊지 않기 위한 기록이다. 시연 후 이 문서만 보고 복원 작업을 시작할 수 있도록 작성한다.

### C.1 무엇이 빠졌나

FMS가 IF-01(ESCORT 요청)을 받았으나 **배정할 로봇이 없을 때**(next_robot이 다른 미션 수행 중 / EMERGENCY / ERROR / 배터리 부족) Interaction에 거절을 알리는 **IF-01 응답 메시지**가 없다. 현재 구조에서 거절은 침묵으로 나타나며, Interaction은 이를 2.2절의 로컬 타임아웃(FMS 무응답과 동일한 증상)으로만 감지한다. 즉 고객은 "시스템 연결이 원활하지 않다"는 안내를 받지, "지금 로봇이 모두 사용 중"이라는 정확한 사유를 받지 못한다.

### C.2 왜 지금은 빼도 안전한가

시연 환경의 전제 때문이다: **동시 고객 1명, 로봇 2대가 해당 미션에 전속.** 이 전제에서 "가용 로봇 없음"은 발생할 수 없는 상태다. 발생 불가능한 실패의 응답 경로를 구현하는 것은 과설계라고 판단했다 (2026.06.06 결정).

### C.3 어떤 조건이 되면 복원해야 하나

다음 중 하나라도 참이 되는 순간 이 보류는 더 이상 안전하지 않다:

- 동시 고객이 2명 이상 가능해짐 (로봇별 독립 호출 허용)
- 로봇이 3대 이상으로 늘어 배정 *선택* 로직이 생김
- 충전(CHARGING) 복귀가 스코프에 들어와 로봇이 미션 불가 상태를 가질 수 있게 됨
- 운영 시간이 길어져 EMERGENCY/ERROR 상태의 로봇과 신규 요청이 공존할 수 있게 됨

### C.4 복원 방법 (설계 초안)

1. **IF-01 응답 메시지 추가** (FMS → Interaction, request_id로 상관):

```json
{
  "request_id": "REQ_0012",
  "result": "ACCEPTED",            
  "mission_id": "MISSION_0012",
  "reject_reason": null,           
  "timestamp": "..."
}
```
   - result: `ACCEPTED` | `REJECTED`, reject_reason: `NO_AVAILABLE_ROBOT` | `ROBOT_ERROR` | ...
   - 부수 효과: Interaction이 mission_id를 알게 되므로 CANCEL을 mission_id로 지정 가능

2. **task 수락 매트릭스 정의** (Robot State × task_type → 수락/거절): 배정 시점과 로봇 상태 변화 사이의 race(예: robot4 배정 직전에 robot4가 다른 고객의 호출을 받아 INTERACTING 전환)는 사전 확인으로 막을 수 없고, 로봇의 task 거절(ACK=REJECT)과 FMS의 재배정/거절 응답으로만 닫힌다. 2.2절 주의사항의 거절 규칙("INTERACTING/ESCORTING 아닌 상태에서 ESCORT 계열 task 거절")이 이 매트릭스의 첫 행이다.

3. **2.2절 타임아웃과의 관계 재정의**: 응답이 생기면 Interaction의 분기가 3개가 된다 — ACCEPTED(안내 시작) / REJECTED(사유 안내: "지금 로봇이 모두 사용 중이에요") / 타임아웃(시스템 장애 안내). 로컬 타임아웃 로직은 삭제하지 않고 최후 방어선으로 유지한다.
