## 1. 시스템 한 줄 요약

지하철 역사(1층/2층, 층마다 Turtlebot4 한 대)의 **안내 로봇 위에 올라가는 터치
키오스크 UI**. 평소에는 순찰하며 웃는 얼굴을 보여주다가, 사용자가 터치/음성으로
시설을 고르면 **IF-01(JSON)을 rosbridge(웹소켓 9090)를 통해 `/information` 토픽으로
발행**하고, 주행 노드(`alfred_driving/web_request_node`)가 이를 받아 릴레이
에스코트를 실행한다. 이동 중에는 "시설 안내중" 화면을 보여준 뒤 끝나면 순찰로
복귀한다. 로봇 쪽 상태 브리지(`alfred_bridge/escort_state_bridge_node` →
**IF-02 `/robotN/ui_state`**)와 YOLO 비상 감지(`/robotN/detection/info`)도 화면을 주도한다.
**별도의 FMS 관제 서버는 없다 — 모든 통신은 rosbridge 서버 하나(허브 노트북, `ws://호스트:9090`)를 거친다**
(서버 설정: `ROSBRIDGE_노트북_매뉴얼.md`).

- 1층 키오스크 = `robot2` / F1, 2층 = `robot4` / F2 (`VITE_FLOOR` 하나로 결정)
- 지원 언어: 한국어 · 영어 · 일본어 · 중국어 (§2.5.3)
- 모드 2종: **일반**(무음 터치) / **시각장애인(VI)** — 웨이크워드로 진입, 전 과정 TTS

## 2. 화면(상태) 8종

| 화면 | 역할 | 진입 조건 |
|---|---|---|
| `patrol` | 순찰 대기 — 풀스크린 웃는 얼굴(#2), TTS 자기소개, 웨이크워드 청취 | 초기 상태 / 모든 복귀 |
| `home` | 초기 메뉴 — A 시설안내·B 음성안내(#3), 층 배지, 언어, 직원호출 | patrol에서 아무 터치 |
| `map` | 건설도면 + 시설 리스트(#4), 층 탭(타층 → 환승 #6) | home에서 A |
| `voice` | STT→LLM 음성안내(#5, §2.5) | home에서 B / 웨이크워드 |
| `guiding` | "시설 안내중" + 진행바(#7) | 시설 확정(로컬) / 로봇 ESCORT_* |
| `waiting` | 사용자 대기 / "N층으로 이동" 환승 안내(#6) | 로봇 WAITING_* / ESCORT_*_FINISHED |
| `charging` | 충전중 + 배터리% | 로봇 DOCKING / UNDOCKING |
| `alert` | 비상경보 전체 점유(ver03) | YOLO FIRE/INJURED/SUSPICIOUS |

직원호출 팝업(#11)은 화면과 **직교하는 오버레이**(staffCallActive)로, 어느 화면 위에도 뜬다.

## 3. 핵심 시나리오 5가지

### 시나리오 A — 터치로 시설 안내 (일반 모드)
1. `patrol`: 웃는 얼굴. 사용자가 화면 터치 → **IF-01 INTERACTING 발행**(로봇 busy 통지) → `home`
2. `home`: [A 시설안내] 터치 → `map`
3. `map`: 도면/리스트에서 시설 탭 → `guideTo()` 실행
   - **IF-01 ESCORT 발행** (poi_id, 층, 출발 좌표, 고객 프로필/언어)
   - 실로봇 연결 시: "안내 준비 중" 화면 → 로봇의 `ESCORT_1F/2F` 수신부터 로봇이 화면 주도
     (로봇이 진행률을 안 주므로 **불확정(움직이는) 진행바**로 표시)
   - 목(mock) 모드: 로컬 시뮬레이션이 진행률을 올림
4. `guiding`: 웃는 얼굴 + "시설 안내중" + 진행바. [취소] → **IF-01 CANCEL** → `patrol`
   (취소 버튼은 **로컬/목 안내에서만** — 로봇 주도 안내는 로봇이 소유)
5. 도착 → `patrol` 복귀 (#10)
   - 목: `arrived` 후 2.6초 / 실로봇: `ESCORT_COMPLETED` 수신 → **"도착했어요!"를 2.6초 표시한 뒤** 복귀

### 시나리오 B — 음성으로 시설 안내 (§2.5)
1. `home` → [B 음성안내] → `voice`
2. 마이크 탭 → STT 청취 (1.5초 침묵 또는 12초 상한에서 자동 종료)
3. LLM이 의도 판정:
   - **facility**: "○○로 안내할까요?" 확인(§2.5.1) → [네] → 시나리오 A의 4번과 동일
   - **chat**: 자유 응답 표시, **서버에 알리지 않음**(§2.5.2) → 재질문 가능
   - **notfound/error**: 재시도 안내
4. 60초 무입력 시 `patrol` 복귀

### 시나리오 C — 시각장애인(VI) 핸즈프리 (웨이크워드)
1. `patrol`: 주기적으로 TTS 자기소개("'헬로 알프레드'라고 말해보세요") — 방송 중엔
   웨이크워드 인식을 꺼서 자기 목소리 오인식 방지
2. "헬로 알프레드" 감지 → **IF-01 INTERACTING**(profile=blind) → `voice` + VI 모드
3. 인사 TTS → 자동 청취 → 결과 낭독 → 시설 확인되면 **자동으로 에스코트 시작**
4. `guiding`: 안내 멘트 TTS + 1초 간격 위치 비프음 → 도착 멘트 → `patrol`(일반 모드로 초기화)

### 시나리오 D — 층간 환승 (#6)
1. 사용자가 타층 시설 선택(예: 1층 키오스크에서 2층 시설) → ESCORT 요청은 동일
2. 이 층 로봇이 엘리베이터/에스컬레이터까지 안내 → 로봇이 `ESCORT_1F_FINISHED` 발행
3. 키오스크는 `waiting`(transfer) 화면: **"2층으로 이동하세요"**
4. 2층 로봇(robot4)이 인계받아 목적지까지 → 2층 키오스크가 `ESCORT_2F` → `ESCORT_COMPLETED`
   (목 모드에서는 같은 흐름을 `awaiting-handoff` 단계로 로컬 시뮬레이션)

### 시나리오 E — 비상경보 (ver03)
1. YOLO(비전 트랙) `detector_node`가 `/robotN/detection/info` 토픽으로 감지 발행 (`event_type` JSON)
2. **어느 화면에 있든** `alert`가 전체 점유 (직원호출 팝업도 강제로 닫음):
   - 화재: 🔥 + 119 사이렌 + "화재발생 화재발생 즉시 안전한 곳으로 대피하십시오"
   - 부상자: 🚒 + (사이렌 없음) + "역사 내 부상자 발생 119에 신고해주세요"
   - 거동수상자: 🚨 + 112 사이렌 + "거동수상자감지 … 잡았다 요놈 잡았다 요놈"
   - 오디오는 "사이렌 → TTS → 쉼 → 반복" 순차 루프 (방송이 항상 또렷함)
3. 경보 중에는 로봇 상태 화면 전환(충전/대기/에스코트)이 **차단**됨
4. 직원이 [경보 해제] → `patrol` 복귀 (해제 탭이 patrol의 "아무 입력→홈"으로 새지 않게 차단)

## 4. 외부 인터페이스 (Interaction 파트의 경계)

| 채널 | 방향 | 토픽 | 상대 노드 / 내용 |
|---|---|---|---|
| IF-01 | 키오스크 → 주행 | `/information` | 수신: `alfred_driving/web_request_node` — `ESCORT`(목적지 확정) · `INTERACTING/STOP`(응대 시작, 순찰 중단) · `CANCEL`(취소) — JSON(std_msgs/String.data) |
| IF-02 | 로봇 → 키오스크 | `/robot2/ui_state`(1층) · `/robot4/ui_state`(2층) | 발행: `alfred_bridge/escort_state_bridge_node` (std_msgs/String, JSON) — `ESCORT_1F/2F` · `WAITING_*` · `ESCORT_*_FINISHED` · `ESCORT_COMPLETED` · `DOCKING/UNDOCKING` · `PATROL` (층별 토픽 격리) |
| 감지 | 비전 → 키오스크 | `/robot2/detection/info`(1층) · `/robot4/detection/info`(2층) | 발행: `alfred_vision/detector_node` — JSON `{event_type: FIRE \| INJURED_PERSON \| SUSPICIOUS_PERSON, …}` |

**별도의 FMS 관제 서버는 없으며**, 세 채널 모두 터틀봇4 허브 노트북에서 도는
**하나의 rosbridge WebSocket 서버(9090)** 를 공유한다 (`createServices.ts`).
코드의 `FmsService`라는 이름은 "IF-01 송신 채널"을 가리키는 관례적 명칭일 뿐이다.
로봇·서버 없이도 콘솔 훅으로 전부 시연 가능: `window.alfredRobotStatus(...)`, `window.alfredAlert(...)`.

## 5. 플로우차트 ↔ 코드 일치 검증 (임무 3)

플로우차트의 **모든 화살표(전이)** 를 코드 위치와 대조한 결과: **전부 구현 일치 ✅**

| 플로우차트 전이 | 이벤트 | 발생 위치 (dispatch) | 처리 위치 (reducer) | 일치 |
|---|---|---|---|---|
| patrol → home (아무 키/터치) | `WAKE` | `PatrolScreen` `useAnyInput` → `enterInteraction` | `kioskMachine` `case 'WAKE'` (patrol에서만) | ✅ |
| patrol → voice (웨이크워드, VI) | `WAKE_VOICE` | `PatrolScreen` `useWakeWord`/버튼 | `case 'WAKE_VOICE'` (mode=VI 설정) | ✅ |
| home → map (A) | `OPEN_MAP` | `HomeScreen` 버튼 A | `case 'OPEN_MAP'` (home에서만) | ✅ |
| home → voice (B) | `OPEN_VOICE` | `HomeScreen` 버튼 B | `case 'OPEN_VOICE'` | ✅ |
| map/voice → home (뒤로) | `GO_HOME` | `MapScreen`/`VoiceScreen` 뒤로 버튼 | `case 'GO_HOME'` | ✅ |
| map/voice → guiding (시설 확정, 목 모드) | `START_GUIDING` | `GuidanceProvider.guideTo` (오프라인 분기) | `case 'START_GUIDING'` | ✅ |
| map/voice → guiding (시설 확정, 실로봇) | `ROBOT_ESCORT(preparing)` | `GuidanceProvider.guideTo` (`env.robotDrivesUi` 분기) | `case 'ROBOT_ESCORT'` | ✅ |
| guiding 진행률 | `UPDATE_PROGRESS` | `MockNavigationService.onProgress` | `case 'UPDATE_PROGRESS'` (세션 없으면 무시) | ✅ |
| guiding → patrol (도착/취소 #10) | `END_GUIDING` | `GuidanceProvider` (도착 2.6초 후 / `cancelGuidance`) | `case 'END_GUIDING'` (전체 초기화) | ✅ |
| home/map/voice → patrol (60초 유휴) | `IDLE_TIMEOUT` | `KioskApp` `useIdleTimer` (3개 화면에서만 활성) | `case 'IDLE_TIMEOUT'` (guiding 보호) | ✅ |
| 직원호출 열기/닫기 (#11) | `OPEN/CLOSE_STAFF_CALL` | 각 화면 `StaffCallButton` / `StaffCallOverlay` | 해당 case (오버레이 토글) | ✅ |
| 어디서든 → alert (ver03) | `DETECTION` | `AlertsProvider`(`/robotN/detection/info` 구독) + `RobotStateProvider`(FIRE 등) | `case 'DETECTION'` (팝업 강제 닫음) | ✅ |
| alert → patrol (직원 해제) | `CLEAR_ALERT` | `AlertScreen` [경보 해제] | `case 'CLEAR_ALERT'` (alert에서만) | ✅ |
| 로봇 DOCKING/UNDOCKING → charging | `ENTER_CHARGING` | `robotStatus.ts` 매핑 → `RobotStateProvider` | `case 'ENTER_CHARGING'` (경보 중 무시) | ✅ |
| 로봇 WAITING_* / *_FINISHED → waiting | `ENTER_WAITING` | `robotStatus.ts` (FINISHED는 transfer+목적층) | `case 'ENTER_WAITING'` | ✅ |
| 로봇 ESCORT_1F/2F → guiding | `ROBOT_ESCORT` | `RobotStateProvider`(poi_id→시설명 해석 후) | `case 'ROBOT_ESCORT'` | ✅ |
| 로봇 ESCORT_COMPLETED → "도착했어요!" → patrol | `ROBOT_ARRIVED` →(2.6초) `END_GUIDING` | `RobotStateProvider`(인터셉트 + 타이머) | `case 'ROBOT_ARRIVED'` / `case 'END_GUIDING'` | ✅ |
| 로봇 PATROL → patrol (남은 경보도 해제) | `EXIT_ROBOT_SCREEN` | `robotStatus.ts` 매핑 | `case 'EXIT_ROBOT_SCREEN'` (로컬 안내·도착 화면 보호, `alert` 포함) | ✅ |

### 요구사항 대비 비고 (검증 중 발견 사항)

1. **#2 문구 차이(경미)**: 요구사항은 "사용할거면 아무키나 누르세요", 구현은
   "사용하려면 화면을 터치하세요"(4개 국어). 터치스크린 기기 특성에 맞게 다듬은
   것으로 기능상 동일. 키보드 입력(`useAnyInput`)도 실제로 동작함.
2. **ver03 부상자 이모지**: 요구사항 "소방차 이모지" → 구현 🚒 (소방차) ✅.
3. **§1.5.1 얼굴 인식 인계**: CV(비전 트랙) 소관으로 UI 파트 범위 밖 — UI는 환승
   화면(waiting/transfer)까지 구현, 미해당.
4. **안전 설계 확인**: 유휴 복귀가 안내중(guiding)을 절대 끊지 않음, 경보 중 로봇
   화면 전환 차단, 로봇 PATROL 수신이 사용자 로컬 안내를 끊지 않음 — 모두 reducer
   가드로 구현 확인.
5. **실연동 토픽 설정**: `escort_state_bridge_node`는 층별 **`/robot2/ui_state` ·
   `/robot4/ui_state`**(std_msgs/String, JSON)에 발행하므로 `.env`의
   `VITE_ROBOT_STATE_TOPIC`은 **비워두면 됨**(`VITE_FLOOR`에서 `/robotN/ui_state` 자동 유도).
   감지는 `VITE_DETECTION_TOPIC=/robot2/detection/info`(2층은 `/robot4/...`)로 지정.
   (`/escort_state`(alfred_interfaces/RobotState)는 FMS·모니터 공용 채널로 **별개** — UI는 안 씀.)
6. **CANCEL 미들웨어 공백**: UI는 취소 시 IF-01 `CANCEL`을 발행하지만
   `web_request_node`는 `STOP/INTERACTING`·`ESCORT`만 처리하고 CANCEL은 무시(warn).
   통합 시 주행 트랙과 협의 필요.
