## 1. `web_request_node.py` — 명령 수신 / 액추에이터

- **수신**: `/information`(IF-01 JSON) · `/robotN/nav_status`
- **출력**: `goal_pose_request` · Dock/Undock 액션 · `stop_request` · `resume_patrol_request`
- **상태머신**: `IDLE → STOPPING → AWAITING_DESTINATION → (TO_TRANSFER → WAITING →) RETURNING → IDLE`

**핵심 함수**
- `information_callback` → `handle_stop`(STOP/INTERACTING: 순찰 즉시 중단) / `handle_escort`(ESCORT: 목적지 검증 후 시작)
- `start_relay`(1F→1F 직행 / 1F→2F 환승), `start_2f_escort`(2F→2F robot4 단독)
- `status_callback`(도착 보고 → 다음 다리), `on_wait_complete`(환승 3초 핸드오프)
- dock/undock 헬퍼: 액션 **완료 콜백까지 대기** 후 다음 단계 (도킹 중 순찰 재개 끼어듦 방지 — 설계 의도 좋음)

**검토 의견**
- ✅ Nav2 비동기 액션을 콜백 체인으로 안전 처리, 서버 미가용 시 폴백 존재.
- ⚠️ **CANCEL 미처리**: UI는 IF-01 `CANCEL`을 보내지만 여기선 STOP/INTERACTING/ESCORT만 처리 → 취소 흐름 미완(통합 시 협의 필요).
- ⚠️ **실패 복구 없음**: 로봇이 `arrived`를 끝내 안 보내면(주행 실패) 상태머신이 멈춤 — 타임아웃/리커버리 경로 부재.

## 2. `escort_state_bridge_node.py` — 상태 관찰 / 번역기

- **수신**: `/information` · `/robotN/nav_status` · `/robotN/detection/info` · `/escort_request`
- **발행**: `/escort_state`(RobotState, FMS·모니터) · `/robotN/ui_state`(String JSON, 키오스크)

**핵심 함수**
- `_on_information`/`_on_nav_status`: **①과 동일한 신호를 보고** 같은 상태 흐름을 복제 → ui_state 생성
- `_on_detection`: `event_type`(FIRE/INJURED_PERSON/SUSPICIOUS_PERSON) → 비상 상태
- `_route_ui`: 상태별로 **어느 로봇의 ui_state 토픽**에 보낼지 라우팅(층 격리)
- `_publish_ui`: **on-change 중복 제거**(직전과 동일하면 발행 생략)

**검토 의견**
- ✅ "관찰자"로 분리해 UI가 주행 로직과 디커플 — 깔끔한 경계.
- ⚠️ **①과 상태 로직 이중 복제(가장 중요)**: 두 노드가 서로 통신하지 않고 각자 같은 신호로 같은 로직을 돌림. **한쪽만 수정하면 UI와 실제 로봇이 어긋남** → 공유 모듈/단일 소스나, 브리지가 ①의 이벤트를 구독하는 구조 검토 권장.
- ⚠️ **on-change의 함정**: 화재 후 상태가 이미 PATROL이면 PATROL을 재발행하지 않아 **UI 경보가 안 닫힐 수** 있음 — "같은 상태 복귀"가 의미 있는 경우 주의(별도 신호 or 강제 발행 고려).

## 3. `RosBridgeRobotStateService.ts` — UI측 수신/파싱

- **역할**: 공유 `RosBridgeClient`(ws://우분투:9090)에 `this.topic`(`/robotN/ui_state`) 구독을 얹고, 들어온 메시지를 `RobotStatusMessage`로 정규화 → `RobotStateProvider`로 전달.

**핵심 함수**
- `onState(handler)`: subscribe → `parseRobotStatus` → 성공 시 로그+handler, 실패 시 `console.debug`로 무시. unsubscribe 반환.
- `parseRobotStatus`/`toSource`: **매우 관대한 파서** — `std_msgs/String`(라벨 `"PATROL"` 또는 JSON 문자열), 커스텀 메시지 필드, snake/camelCase 모두 수용.

**검토 의견**
- ✅ 방어적 파싱으로 **로봇이 어떤 포맷으로 보내든** 흡수 → 통합 마찰 최소화(실전에서 토픽 포맷 불일치를 잘 버팀).
- ✅ 수신/파싱/구독해제가 한 파일에 응집, 화면 로직과 분리.
- ⚠️ **조용한 무시**: `state`를 못 찾으면 `console.debug`(기본 숨김)로 버림 → 잘못된 메시지가 운영 중 묻힐 수 있음. 디버깅 시 콘솔 레벨 Verbose 필요(또는 warn 승격 고려).
- 참고: 진행률(`progress`)은 ②가 안 보내므로 항상 비어 옴 — UI는 불확정 바로 처리(별개 결정).

## 4. `proxy.mjs` — 음성 백엔드 (키 격리)

- **엔드포인트 3개**: `POST /api/llm/understand`(Claude 구조화 출력으로 시설/잡담 판정), `POST /api/soniox/temporary-api-key`(단기 임시키 발급), `GET /api/health`.
- **핵심**: `ANTHROPIC_API_KEY`/`SONIOX_API_KEY`는 **이 프로세스에만**. 브라우저는 `/api`로만 호출(임시키만 받음).

**검토 의견**
- ✅ 키 격리 설계 적절(브라우저 번들에 비밀 미포함). 구조화 출력(zod)으로 파싱 안정성 확보.
- ✅ 입력 검증·에러 코드(400/500/502) 명확.
- ⚠️ **인증/레이트리밋 없음**: 키오스크 LAN 전제라면 OK이나, 외부 노출 시 보강 필요.

## 5. 종합 리뷰 포인트

**강점**
1. **관심사 분리가 명확** — 액추에이터(①) / 관찰자(②) / 수신파서(③) / 키격리(④)로 책임이 깔끔히 갈림.
2. **UI ↔ 주행 디커플** — 토픽·JSON 계약으로만 연결, rosbridge 단일 허브로 수렴(별도 FMS 서버 없음).
3. **통합 친화적 파싱·발행** — 관대한 수신 파서 + on-change 발행 + 비상 채널 분리.

**우선 검토 권장(리스크 순)**
1. **상태머신 이중 복제(①↔②)** — 동기화가 암묵적. 한쪽 수정 시 어긋남 위험 → 단일 소스화 검토. *(영향도 高)*
2. **on-change dedup + 비상 해제** — "같은 상태 복귀" 미발행으로 경보가 안 닫힐 수 있음. *(影响 中)*
3. **CANCEL 흐름 미완(①)** — UI 취소가 로봇까지 전달 안 됨. *(中)*
4. **조용한 무시(③) / 실패 복구 부재(①)** — 운영 가시성·복원력 보강 여지. *(低~中)*
